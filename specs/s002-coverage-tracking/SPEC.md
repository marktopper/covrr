# Spec S002: Coverage Tracking

**Task:** MAR-113
**Parent:** MAR-111 (Plan Covrr)
**Status:** Draft
**Created:** 2026-04-24

---

## Summary

Covrr collects coverage data from Playwright test runs and stores it per version, enabling historical tracking and delta reporting. Coverage is emitted by user scripts via a stdout protocol, stored in the local state directory, and compared between versions.

---

## 1. Overview

### What "coverage" means in Covrr

Coverage in Covrr refers to **code coverage** — the fraction of production code exercised by tests. This is distinct from test pass rate (which is handled by the Playwright runner natively).

Covrr supports any coverage tool that can emit JSON coverage reports (Istanbul/NYC for JS/TS, Coverage.py for Python, etc.). Covrr does not generate coverage itself — it collects, stores, and compares outputs generated by user-installed tools.

### Coverage flow

```
User's test runs → coverage tool emits JSON → Covrr parses → stored per version → diff computed
```

---

## 2. Coverage Data Format

### Coverage report schema

When a Playwright script emits coverage data, it writes a JSON file (path configured in `covrr.yaml`) or stdout a line matching the coverage protocol.

```typescript
interface CoverageReport {
  version: string;           // Version string (from S003)
  script_name: string;       // Which script generated this
  generated_at: string;      // ISO 8601 timestamp
  tool: string;              // e.g. "istanbul", "nyc", "coverage.py"
  summary: {
    lines_total: number;
    lines_covered: number;
    lines_percent: number;
    branches_total: number;
    branches_covered: number;
    branches_percent: number;
    functions_total?: number;
    functions_covered?: number;
    functions_percent?: number;
  };
  files?: CoverageFile[];   // Optional per-file breakdown
}

interface CoverageFile {
  path: string;             // Relative file path
  lines_total: number;
  lines_covered: number;
  lines_percent: number;
}
```

### Emitting coverage from scripts

Users configure their coverage tool to output JSON. Covrr reads the configured output path after each script runs.

**Configuration in `covrr.yaml`:**

```yaml
coverage:
  tool: istanbul  # or nyc, coverage.py, etc.
  output_path: ./coverage/coverage-final.json  # absolute or relative to covrr.yaml
  # OR: emit via stdout (scripts print COVERAGE:<base64-json>)
```

### Stdout protocol

For cases where a coverage file isn't practical, scripts can emit coverage via stdout:

```
COVERAGE:<base64-encoded-json>
```

Covrr reads stdout after the Playwright process exits and looks for lines matching `^COVERAGE:.*$`. The first match is parsed as the coverage report.

## 3. Storage

### Location

Coverage reports are stored in the Covrr state directory:

```
.covrr/
├── state/
│   └── coverage/
│       ├── <version>/
│       │   ├── <script-name>.json    # one file per script
│       │   └── manifest.json         # version-level summary
│       └── latest -> symbolic link to most recent version
```

### Version directory

Each version that has coverage data gets a directory named after the version string (sanitized for filesystem safety: alphanumeric + dash/underscore).

### `manifest.json` per version

```json
{
  "version": "1.2.3",
  "captured_at": "2026-04-24T10:00:00Z",
  "scripts": ["smoke", "e2e", "coverage"],
  "total_lines": 5000,
  "total_lines_covered": 4500,
  "overall_percent": 90.0
}
```

### Version detection

Version is provided by S003 (Version Change Detection). Covrr receives a version string when invoking scripts and tags the coverage data with it.

**If version is unknown at time of run** (e.g., no git tag, no package.json):
- Use `git rev-parse HEAD` as fallback version identifier
- If not a git repo, use `datetime.now().isoformat()` as a timestamp-based version

## 4. Coverage Thresholds

### Threshold definition

In `covrr.yaml`:

```yaml
coverage:
  thresholds:
    lines_percent: 80      # minimum 80% line coverage
    branches_percent: 75   # minimum 75% branch coverage
    # Both must pass for the run to be considered "passing"
```

### Enforcement

After each script run, Covrr:
1. Loads the threshold config
2. Compares the coverage report against thresholds
3. If any threshold is breached → marks script as `failed_threshold`
4. Emits a warning to stdout: `Coverage below threshold: lines {actual}% < {threshold}%`

### Threshold failure vs test failure

These are independent:
- A script can pass Playwright tests but fail coverage threshold (warning only, not hard fail unless `strict: true`)
- A script can fail Playwright tests regardless of coverage

```yaml
coverage:
  strict: true  # if true, threshold breach causes non-zero exit code
```

## 5. Delta / Comparison

### Comparing two versions

Covrr provides a `covrr diff <from-version> <to-version>` command (see S004 for output format) that shows which files/lines gained or lost coverage between two versions.

### Delta calculation

```
For each file in the newer version's coverage:
  - If file existed in older version: delta = new_percent - old_percent
  - If file is new: delta = new_percent (positive — new code covered)
  - If file was removed: excluded from delta

Files with >5% coverage drop are flagged as "degraded".
```

### Storage for delta

Deltas are computed on-demand from stored coverage reports. No extra storage needed beyond the per-version JSON files.

## 6. CLI Interface

### `covrr coverage report [--script <name>] [--version <ver>]`

Outputs a coverage summary for the given version/script.

```bash
covrr coverage report --version 1.2.3 --script smoke
# Lines: 85% (5100/6000) | Branches: 78% (1200/1540) | Tool: istanbul
```

### `covrr coverage diff <from> <to>`

Shows coverage delta between two versions.

```bash
covrr coverage diff 1.2.2 1.2.3
# Lines: +2.1% (83.4% → 85.5%) | Branches: -0.5% (78.5% → 78.0%)
# Degraded files: src/utils/auth.ts (-8.2%)
```

### `covrr coverage status`

Shows current version's coverage status against thresholds.

```bash
covrr coverage status
# smoke: PASS (lines 87%, above threshold 80%)
# e2e: PASS (lines 91%, above threshold 80%)
# coverage: FAIL (lines 71% < threshold 80%, strict mode)
```

- The `output_path` in `covrr.yaml` can reference script-specific overrides
- Coverage data is stored keyed by script name + version

## 8. Edge Cases

| Situation | Behaviour |
|-----------|-----------|
| Coverage tool not installed | Warning: "Coverage tool 'istanbul' not found. Skipping coverage collection." — script still runs |
| Coverage output file missing | Warning: "Coverage file not found for script '<name>'. Skipping." |
| Invalid JSON in coverage file | Error: "Coverage file corrupt for '<name>': <parse error>" — script marked failed |
| stdout has no `COVERAGE:` line | Fall back to file-based collection. If neither found → warn and skip |
| Version directory already exists | Merge — scripts that ran again overwrite, new scripts appended |
| Threshold file missing | Default: no threshold enforcement (warn if `strict: true` but no thresholds defined) |
| Git not available for version fallback | Use timestamp-based version identifier with warning |

## 10. Dependencies

- Coverage tools are **peer dependencies** — users install their own (istanbul, nyc, coverage.py)
- `js-yaml` for config parsing
- `zod` or `joi` for coverage schema validation (optional but recommended)
- Built with Node.js (TypeScript)
